# use-herald-action

GitHub Action that adds comments, reviewers, assignees, labels, and commit statuses to
pull requests based on "Herald" rule JSON. Written in TypeScript, bundled to a single
committed JS file that GitHub runs directly at consumer runtime (no build step for
consumers of the action).

## Commands

```
npm install                      # deps (also run by pre-commit on package.json change)
npm run build                    # esbuild bundle -> dist/index.js  (REQUIRED after src changes)
npm run test                     # jest, all *.spec.ts, coverage on
npx jest __tests__/rules.spec.ts # single test file
npx jest -t "pattern"            # single test by name
npx jest -u                      # update snapshots
npm run lint                     # eslint index.ts
npx pretty-quick                 # prettier format
npx ts-auto-guard src/rules.ts   # regenerate src/rules.guard.ts
```

## Critical gotchas

- **Rebuild the bundle after editing `src/`.** `action.yml` runs `dist/index.js`
  (`using: node20`). The pre-commit hook (husky + lint-staged) rebuilds and re-stages
  `dist/` automatically on any `.ts` commit, but if committing outside that hook, run
  `npm run build` and commit `dist/` yourself. CI (`use_action_for_build_test.yml`)
  dry-runs the action with the committed `dist/`, then rebuilds and dry-runs again —
  it fails if the committed bundle is stale.
- **Never hand-edit `src/rules.guard.ts`** — it's generated by `ts-auto-guard` (header
  says so). Change `src/rules.ts` then regenerate with `npx ts-auto-guard src/rules.ts`.
  There's no npm script for this; it's manual.
- Tests are `*.spec.ts` (not `.test.ts`) in `__tests__/`. GitHub API calls are stubbed
  with `nock` using fixtures under `__mocks__/scenarios/`, event payloads under
  `__mocks__/*.json`, and sample rule files under
  `__mocks__/{rules,required_rules,dispatch_rules}/`.
- Node: `.nvmrc` says 16.13.1 but the action runtime and esbuild target is node20
  (`engines.node >=20.0.0`). CI step labels say "Node.js 12.x" — that's a stale label;
  the steps actually read `.nvmrc`.

## Architecture map

- `index.ts` → `src/index.ts` `main()`: reads action inputs, loads the event JSON
  (`GITHUB_EVENT_PATH`), loads rules, diffs the PR (`compareCommits`), matches rules,
  dispatches action handlers grouped by action, sets the `appliedRules` output. All
  wrapped in try/catch → `setFailed`.
- `src/rules.ts` — the rule engine core. `Rule` interface, `RuleActions`/`RuleMatchers`
  enums, `Rules.loadFromLocation()` (fast-glob + validate + sanitize each rule file),
  `MatchingRules.load()` (matches rules concurrently via `p-queue`). A rule matches only
  if **all** of its present matchers pass; `status` rules always run regardless of
  match outcome, so they can report failure.
- Matchers (`RuleMatchers`): `includes`/`excludes` (minimatch glob against changed
  filenames), `includesInPatch` (regex against diff patch content), `eventJsonPath`
  (JSONPath query against the event payload — the only matcher usable for
  `repository_dispatch`, since that event has no PR to diff).
- Action handlers, one per `RuleActions` value, all implementing `ActionMapInput`:
  `src/comment.ts`, `src/reviewers.ts`, `src/assignees.ts`, `src/labels.ts`,
  `src/statuses.ts`. `src/index.ts` groups matched rules by action and runs the
  handlers in parallel via `actionsMap`.
- Central types: `Event`, `RuleFile`, `SUPPORTED_EVENT_TYPES` in
  `src/util/constants.ts`; `Rule`/`MatchingRule` in `src/rules.ts`;
  `ActionInput`/`ActionMapInput` in `src/index.ts`.
- Supported events: `pull_request`, `pull_request_target`, `push`,
  `repository_dispatch`. Dispatch events skip the PR diff and action dispatch entirely
  — they only produce matches via `eventJsonPath`, surfaced through the `appliedRules`
  output.

Rule JSON syntax and end-user documentation live in `README.md`, not here.

## Conventions

- Prettier: single quotes, semicolons, `printWidth: 120`. ESLint:
  `@typescript-eslint/recommended` + prettier config. TypeScript `strict: true`.
- Adding a new action handler: implement `ActionMapInput`, register it in `actionsMap`
  (`src/index.ts`), add the value to the `RuleActions` enum (`src/rules.ts`),
  regenerate guards (`npx ts-auto-guard src/rules.ts`), and add a `*.spec.ts` with
  `nock` fixtures under `__mocks__/scenarios/`.
